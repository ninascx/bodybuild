# 全站健康检查

本项目的健康检查优先保护三件事：数据可信、训练不中断、复盘不误导。UI 美化和动效收敛排在这些之后。

## 检查命令

- `npm run health:check`：轻量健康检查，覆盖 metrics、导出、PWA 配置和 UI 回归约束。
- `npm run test`：lint + typecheck。
- `npm run bundle:check`：检查已构建前端资源的 gzip 体积预算，需先执行 `npm run build`。
- `npm run export:check`：导出 JSON/CSV/摘要内容检查。
- `npm run build`：生产构建和服务端打包。
- `npm run prod:check`：生产启动、认证、备份和基础 API 检查。

## 五个维度

### 数据可靠性

- 保存流程必须先落本地缓存，再同步服务端。
- 保存失败要明确进入离线/可重试状态，不能让用户误以为数据丢失。
- 导出必须覆盖每日记录、训练组、有氧、模板和用户配置。
- 服务端空但本地有数据时，优先保护本地数据。

### 移动端训练

- 训练模式的首要任务是不中断：当前动作、当前组、上一/下一动作、休息计时、加减组都要可达。
- 键盘打开时底栏不能遮住当前输入。
- 休息结束提醒不能长期闪烁抢注意力。
- 结束训练必须保留已记录组，并同步今日记录。

### 复盘统计

- 体重、热量、蛋白、腰围要处理稀疏数据和空状态。
- 训练表现按真实动作统计，不按固定动作名硬匹配。
- 自定义动作改名后应按 `exerciseId` 合并历史。
- 有氧暂不混入力量表现折线，避免同轴误读。

### PWA/缓存

- Service Worker 不缓存 `/api/*`。
- 新版本提示要说明重新加载不会影响已保存数据。
- 旧缓存清理要避免旧 bundle 导致白屏或旧 UI 残留。
- PWA 安装后应保留主题、安全区和基础离线能力。

### UI 一致性

- 移动端不允许横向溢出。
- 避免任意 `z-[...]`、手写长动画、无限 pulse。
- 固定底栏必须处理 `safe-area-inset-bottom`。
- 菜单、图例、动作名等长文本要截断或换成可读布局。

### 性能预算

- 主入口 JS gzip 预算当前为 108 KiB，阶段目标已收敛到 110 kB 以下。
- 每日记录、训练、分析外壳、周报、设置、管理、导出弹窗必须保持独立 chunk。
- Recharts 只应进入趋势图表 chunk，不进入主包或分析外壳。
- 调整 tab 页面、导出弹窗或图表依赖后，必须跑 `npm run build` 和 `npm run bundle:check`。

## 当前基线
